# 読書レビュー -- PDFから要約・壁打ち・記事生成

本のPDFを読み込み、要約→感想の壁打ち→記事生成までを対話的に進める。
本を読むのが苦手なユーザーが、要約を読むことで本の内容を効率的にインプットし、対話を通じて人生に活かすアクションまで落とし込むためのスキル。

## 使い方

```
/book-review books-pdf/本のファイル名.pdf
/book-review books-pdf/本のフォルダ名/
```

## 前提条件

- PDFファイルが`books-pdf/`ディレクトリに配置されていること
- `books-pdf/`はgitignoreされているため、ローカルのみに保存される
- **1ファイル100MB以下**に分割すること（大きいと読み取りがタイムアウトする）
- 分割PDFは本ごとにサブディレクトリにまとめる

## 全体の流れ

```
STEP 1: PDF読み込み → 本の情報確認
STEP 2: 要約を箇条書きで生成 → ユーザーが読む（= 読書完了）
STEP 3: ユーザーの感想を受け取る
STEP 4: 壁打ち対話（コーチング）
STEP 5: 記事を生成して保存
```

---

## STEP 1: PDFの読み込みと確認

引数で渡されたPDFファイル（またはフォルダ内の全PDF）を読み込み、以下を把握する:

1. 本のタイトル、著者名
2. 目次・章構成
3. 各章の主要な内容

分析結果をユーザーに提示し、以下を確認する:

- 本のタイトル（正確か）
- 著者名（正確か）
- 本の購入リンク（Amazon等のURL）
- タグ（カテゴリ。例: 自己啓発、習慣、ビジネス、技術。AIに任せてもOK）

さらに以下を質問する:

- **「なぜこの本を手に取りましたか？（あなたの課題は何ですか？）」**
- **「この本から得たい情報は何ですか？」**

## STEP 1.5: 他の人の感想をリサーチ

本のタイトルと著者名をもとに、Web上から他の読者の感想・レビューを10件程度収集する。

**検索方法:**

- WebSearchで「{本のタイトル} 感想」「{本のタイトル} レビュー」「{本のタイトル} 書評」を検索
- ブログ記事、note、読書メーターなどから多様な意見を収集する
- 肯定的な感想だけでなく、批判的な意見も含める

**出力形式:**

各感想は**200文字程度**で記載する。1行の要約ではなく、読者がどう感じたかの文脈が伝わる分量にする。

```
## 他の読者の感想

### 1. 感想のタイトル（1行）
200文字程度の感想内容。どのような背景の読者が、何に共感し、何を学んだかが分かるように書く。
肯定的な意見だけでなく、批判的な感想も含める。
[ソース名](URL)

### 2. 感想のタイトル
...
```

- 各感想は**200文字程度**で書く（表形式ではなくリスト形式）
- ソースURLを必ず付ける
- 10件程度を目安に収集する
- 肯定的な感想だけでなく、批判的な意見も含める
- ユーザーが読む前に「他の人はこう感じている」という多角的な視点を提供する

収集結果はSTEP 2の要約と一緒にユーザーに提示する（別々に出さない）。

## STEP 2: 要約の生成

ユーザーが要約を読むこと = 読書完了となるため、本の核心を効率的に伝えることが目的。

**STEP 1.5のリサーチ結果 + 要約をまとめて1回で提示する。** 以下の順序で出力する:

1. 他の読者の感想（引用形式）
2. 全体の要約
3. パートごとの要約（200文字の要約文 + 要点の箇条書き + 具体的アクション）

### 要約のフォーマット

章の分け方は本の構造に合わせて柔軟に判断する（パート単位、章単位など）。各パート/章ごとに以下の形式で要約する:

```markdown
### パートX: パートタイトル

200文字程度の要約文。このパートが何を伝えようとしているのか、前後の文脈も含めて理解できる分量で書く。

**要点:**

- ポイント1
- ポイント2
- ポイント3

**具体的なアクション:**（本に記載がある場合のみ）

- アクション1
- アクション2
```

**重要:**

- 要約文は200文字程度で、パートの核心が伝わるように書く。短すぎて理解できないのはNG
- 要点は箇条書きで分かりやすく
- 本の中に具体的なアクションや実践方法が記載されている場合は必ず「具体的なアクション」として書き出す

**要約はまだ記事に保存しない。まずユーザーに読んでもらう。**

要約を提示した後、以下のメッセージを表示する:

```
要約は以上です。読み終わったら感想を教えてください。
どんなことでもOKです（印象に残ったこと、共感したこと、疑問に思ったことなど）。
```

## STEP 3: 感想の受け取り

ユーザーが感想を返したら、そのまま壁打ちフェーズに入る。

## STEP 4: 壁打ち（コーチング対話）

### あなたの役割

このステップでは、あなたは**ユーザーの読書コーチ**として振る舞う。

**重要: 回答は自分が生成した要約からではなく、以下の情報源を使って行う:**

1. **本のPDFデータ**: STEP 1で読み込んだPDFの原文を参照し、該当する箇所を引用（`>`）しながら回答する
2. **Web検索**: 本の内容に関連する実践事例、研究データ、他の専門家の見解をWebSearchやWebFetchで調べて回答に含める
3. **関連書籍・理論**: 同じテーマの他の本や理論を引き合いに出し、多角的な視点を提供する

要約はあくまでユーザーが読むための「本の代わり」であり、コーチングの回答は本の原文とリアルタイムのリサーチに基づく。

### 壁打ちの観点

1. **課題との接続**: STEP 1で聞いた「なぜこの本を手に取ったか」「得たい情報」と、本の内容を結びつける
2. **人生へのプラス**: この本の知見が日常生活・仕事・キャリアにどうプラスになるかを具体的に提案
3. **ネクストアクション**: 明日からできる具体的な行動を一緒に考える
4. **習慣化**: 一度きりではなく継続的な習慣にするにはどうすればいいかを提案
5. **振り返り時期**: 成果をいつ振り返るか（1週間後、1ヶ月後など）を設定する

### 壁打ちのスタイル

- 一方的にアドバイスするのではなく、**質問ベース**で対話を進める
- ユーザーの言葉を受け止めた上で、深掘りする質問を投げかける
- 回答には**本の原文からの引用**（`>`表示 + ページ番号）を含める
- 関連するWebの情報も調べて補足する
- 具体的なアクションに落とし込むまで対話を続ける
- ユーザーが「OK」「これでいい」と言ったらSTEP 5に進む

### 壁打ちの質問例

- 「この中で、明日から実際にやれそうなことはありますか？」
- 「今の仕事や生活で、この考え方が一番効きそうな場面はどこですか？」
- 「1ヶ月後に振り返るとしたら、何が変わっていたら成功ですか？」
- 「習慣にするために、トリガー（きっかけ）になりそうなことはありますか？」

## STEP 5: 記事の生成

壁打ちが完了したら、対話の内容をもとに記事を生成する。

**ファイル名**: `books/NNNN_slug.md`（NNNNは連番、slugは本のタイトルの英語スラッグ）

**日時**: `TZ=Asia/Tokyo date "+%Y-%m-%d %H:%M"`で取得したJST

**連番**: `books/`ディレクトリ内の既存ファイルから次の番号を自動計算

```markdown
---
id: { 連番 }
title: '{本のタイトル}'
author: '{著者名}'
bookUrl: '{購入リンク}'
tags: ['{タグ1}', '{タグ2}']
rating: { 1-5 }
readDate: '{YYYY-MM-DD}'
create: '{YYYY-MM-DD HH:MM}'
---

## この本について

> [{本のタイトル}]({購入リンク})

{本の概要を2-3行で}

## 他の読者の感想

### 1. {感想タイトル}

> {200文字程度の感想内容。引用表示にする}

[{ソース名}]({URL})

### 2. {感想タイトル}

> {200文字程度の感想内容。引用表示にする}

[{ソース名}]({URL})

（10件程度繰り返す）

## なぜこの本を手に取ったか

{STEP 1でユーザーが答えた課題}

## この本から得たかった情報

{STEP 1でユーザーが答えた得たい情報}

## 要約

### 全体のまとめ

- ポイント1
- ポイント2
- ポイント3

### 第1章 章タイトル

- ポイント1
- ポイント2
- ポイント3

（以降、各章ごとに繰り返し）

## 感想

{STEP 3-4の対話で出たユーザーの感想をまとめる}

## ネクストアクション

- [ ] {具体的なアクション1}（期限: YYYY-MM-DD）
- [ ] {具体的なアクション2}（期限: YYYY-MM-DD）
- [ ] {具体的なアクション3}（期限: YYYY-MM-DD）

## 振り返り予定

- {振り返り日}: {何を確認するか}

## 読書メモ

（後から追記するセクション）
```

### 記事生成後

1. `npm run dev`が起動していなければ起動する
2. `http://localhost:3000/note/books/NNNN_slug/` のURLをユーザーに提示
3. ユーザーがlocalhostで確認できるようにする
4. **通知を送る**こと

## 記事生成後の追記サポート

スキル実行後、ユーザーが以下を依頼した場合に対応する:

- 「感想を追記」→ 感想セクションに追記
- 「アクション完了」→ ネクストアクションのチェックボックスを更新
- 「振り返りメモ」→ 振り返り予定セクションに結果を追記
- 「読書メモ追加」→ 読書メモセクションに追記

## 注意事項

- **PDFの内容をそのまま記事にコピーしない**。著作権に配慮し、要約と自分の言葉での解釈に留める
- **本の内容を引用する場合は、必ずMarkdownの引用表示（`>`）を使用する**。引用元のページ番号も可能な限り付記する
  ```markdown
  > 「どうしてこんなに大変なんだ」ではなく「どうすればもっと簡単になるか」と考える。
  > -- p.35
  ```
- **PDFファイルはgitにコミットしない**。`books-pdf/`はgitignore対象
- 本の購入リンクは**必ず記事の冒頭に配置**する
- 要約は**箇条書き**で書く。長文にしない
- 壁打ちフェーズでは**ユーザーの合意なく記事を生成しない**。対話が完了してから保存する
- 記事の評価（rating）と読了日（readDate）は壁打ち完了時の日付で記入する
